# Инструкция исполнителю — доработка документации lsFusion

Привет! Мы поблочно дорабатываем документацию lsFusion (tier'ы **Paradigm** и **Language**, EN+RU). За тобой закреплён **один тематический трек** (домен, напр. «Действия» или «Формы») — так ты не прыгаешь между темами. Внутри трека блоки идут снизу вверх; тебе выдают **один блок** за раз и после сдачи — следующий блок того же трека.

## Что ты получаешь

Оркестратор присылает тебе **готовый промпт блока** (из `DOCS-DESC-PROMPTS.md`) — например «B05 — Арифметика/сравнение/логика». В промпте уже перечислены **конкретные файлы**, которые надо доописать (пары `docs/<type>/en/*.md` + `docs/<type>/ru/*.md` — структура type-first: paradigm-статьи лежат в `docs/paradigm/`, language — в `docs/language/`). Скармливаешь этот промпт своему ИИ-агенту (Claude Code или codex) в корне репозитория и работаешь над блоком **в выданной под блок ветке**. Если конструкции блока ещё нет статьи — **создай** обе версии (`docs/<type>/en/<имя>.md` + `docs/<type>/ru/<имя>.md`) и сам подключи их в навигацию `docs/sidebars.js` тем же блочным коммитом (`sidebars.js` — часть документации).

Перед блоком сверься с разделом **«Полнота против грамматики»** в `DOCS-DESC-BLOCKS.md`: если твой блок там упомянут — обязательно покрой перечисленные опции **внутри** статей, а помеченные там **новые статьи создай**.

## Что прочитать до старта (один раз)

Тебе (человеку) `AGENTS.md` и `DOCS-DESC-PLAN.md` читать не нужно — это файлы **для ИИ-агента**, промпт блока сам велит их прочитать. Твоя инструкция — этот файл. Достаточно понимать процесс ниже и пинговать оркестратора по готовности/вопросам.

## Жёсткие правила (нарушение = блок вернут)

1. **Блок идёт в два под-цикла: сначала ЦЕЛИКОМ Paradigm, потом ЦЕЛИКОМ Language.** Каждый под-цикл — это описание + свой коммит + завершённое ревью. К Language переходишь **только** когда под-цикл Paradigm закрыт полностью. Paradigm — абстракция (как будто синтаксиса и расширений нет); Language — синтаксис и механика именно той абстракции, что уже изложена в Paradigm (ссылается на неё, не переписывает).
2. **Не смешивай tier'ы.** Синтаксис не валим в Paradigm; абстракцию не валим в Language.
3. **Источник истины — код платформы**, не память и не плагин: `LsfLogics.g` → `ScriptingLogicsModule.java` → `LogicsModule.java` → классы; примеры в `server/src/main/lsfusion`. Факты, которые легко переврать (NULL-propagation, классы результата, scope событий/PREV, потокобезопасность) — проверяй по Java/codex.
4. **EN и RU — построчно синхронны:** одна структура и один состав информации, это переводы одного текста.
5. `AGENTS.md` (правила для ИИ) **можно дополнять**: нашёл общее правило/проблему — твой агент дописывает его в `AGENTS.md` **отдельным коммитом** (не в блочном). Это файл для ИИ, людям читать не надо. `CLAUDE.md` не трогай — это оркестратор. А вот навигацию `docs/sidebars.js` обновляй сам в блочном коммите (новые / перемещённые / переименованные статьи) — `sidebars.js` часть документации.
6. Каждый блок ведётся **в своей ветке** от `master` (`docs/<block-id>`, напр. `docs/B05`) — **создай её первым действием и работай в отдельном рабочем дереве (`git worktree add`), не в общем чекауте**: при параллельных блоках несохранённые правки в одном каталоге молча затираются друг другом. Коммить в ветку рано (commit-first). Прямо в `master` не коммить: блок попадает в `master` только после закрытия обоих под-циклов, и **merge делает оркестратор**.

## Как сдавать (commit-first, по под-циклам)

Каждый под-цикл сдаётся отдельно — сперва Paradigm целиком, затем Language целиком.

1. Доделал tier → **закоммить** (`docs: <block-id> paradigm — <заголовок>`, потом отдельным коммитом `docs: <block-id> language — <заголовок>`).
2. Прогони **self-review** (явная сверка с каждым релевантным правилом AGENTS.md; полнота покрытия по коду; EN/RU parity; направление ссылок; чистота tier'ов) **и codex review**. Каждое замечание чини **отдельным коммитом**, повторяй, пока проход не станет чистым. Только после чистого Paradigm-под-цикла начинай Language.
3. Когда оба под-цикла чистые — **запушь ветку блока** (`docs/<block-id>`) и напиши оркестратору. Он смотрит ветку и **мерджит её в `master`** (squash по необходимости). Пушишь только свою ветку блока (для ревью); сам в `master` ты не мерджишь и не пушишь.
4. Если есть замечания — оркестратор их пришлёт, ты фиксишь отдельными коммитами и снова сдаёшь.

## Чего НЕ делать

- Не описывай внутренние оптимизации, extension-механику внутри базовых статей, deprecated-функционал.
- Не изобретай новых слов, когда есть обычные; в сходных конструкциях — единая терминология.
- Не пиши метафор и импортированных лейблов из других технологий — только платформенные термины буквально.
- Не отмечай статусы в `DOCS-DESC-BLOCKS.md` — реестр ведёт **только оркестратор**.

Вопросы — в тред. Спасибо!
