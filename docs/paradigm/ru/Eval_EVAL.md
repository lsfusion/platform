Оператор выполнения программного кода создает [действие](Actions.md), которое во время работы сервера компилирует строку с исходным кодом на языке lsFusion и выполняет её внутри сервера с полным доступом к остальному проекту.

### Исходный код

Строка трактуется как тело свежего [модуля](Modules.md), сгенерированного платформой. Заголовок модуля платформа формирует автоматически: уникальное имя модуля плюс зависимости от всех загруженных на этот момент модулей. Поэтому код может ссылаться на любой [элемент](Element_identification.md) проекта и сам объявлять новые локальные элементы.

Принимаются две формы исходного кода — в зависимости от режима оператора:

-   *Режим инструкций* (по умолчанию). Строка является последовательностью инструкций. Одна из них должна создавать действие с именем `run` — именно оно и будет выполнено. Как обычное скриптовое действие, `run` может объявлять интерфейсные параметры и получать через них значения.
-   *Режим действия*. Строка трактуется как тело действия — последовательность [операторов-действий](Action_operators_paradigm.md) и объявлений локальных свойств, — которое платформа автоматически оборачивает в действие `run`. Тело обращается к runtime-аргументам eval-действия позиционно.

### Параметры

Действие, выполняющее программный код, принимает исходный код как свой первый параметр. Дополнительные значения, передаваемые оператору, становятся runtime-аргументами `run` при его вызове и подчиняются обычным правилам передачи параметров.

### Область выполнения

Скомпилированное действие `run` выполняется синхронно в [сессии](Change_sessions.md) вызывающего действия, поэтому его чтения и записи разделяют сессию, apply и транзакционное состояние с вызывающим.

### Ограничения на выполняемый код

Выполняемый код предназначен для добавления новых локальных элементов и вызова существующих; постоянную структуру проекта он менять не может. В коде eval не могут появляться:

-   объявления новых [классов](Classes.md), [таблиц](Tables.md) и [индексов](Indexes.md)
-   глобальные [события](Events.md) и [ограничения](Constraints.md), включая формы write-события и follows-ограничения, привязанные к объявлению свойства
-   [агрегации](Aggregations.md), а также режимы агрегации и равенства у [группировки](Grouping_GROUP.md)
-   [материализация](Materializations.md), явное размещение в таблице и not-null ограничения на свойствах
-   объявление [группы](Groups_of_properties_and_actions.md) с родительской группой из другого модуля или привязка свойства или действия к группе из другого модуля
-   добавление базовых событий к существующим действиям

[Data-свойства](Data_properties_DATA.md), объявленные внутри eval-кода, приводятся к области сессии; постоянные data-объявления не допускаются.

### Язык

Для создания действия, выполняющего программный код, используется [оператор `EVAL`](../language/EVAL_operator.md).

### Примеры

```lsf
// тело действия задано напрямую
addProperty { EVAL ACTION 'MESSAGE \'Hello World\''; }

// произвольный скрипт, введённый пользователем
code 'Исходный код' = DATA BPSTRING[2000] ();
execute 'Выполнить код' { EVAL code(); }

// передача runtime-аргумента в выполняемое тело в режиме действия
greet 'Поприветствовать пользователя' (CustomUser u) { EVAL ACTION 'MESSAGE \'Здравствуй, \' + name($1);' PARAMS u; }
```
